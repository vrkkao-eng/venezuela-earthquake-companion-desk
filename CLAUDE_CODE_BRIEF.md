# Claude Code 工作指令:Venezuela Earthquake Companion Desk

## 0. 背景(請先讀完才開始動工)

這不是練習專案。EMSR884 是 2026 年 6 月 24 日委內瑞拉 7.5 級地震後 Copernicus EMS 真實啟動的快速製圖代碼,目前已有破百人死亡。你正在接手的程式碼最終可能被前線救援團隊用來規劃路徑、判斷建築物危險程度。**任何「看起來在運作但其實是假數據/假邏輯」的程式碼,危害比「還沒做」更大**,因為使用者會誤以為這是真實計算結果。

你會收到兩份輸入檔案:
- `legacy/fragment_zh.html`(舊草稿,只有部分模組,中文 UI)——**僅供參考,不要以此為基礎開發**
- `legacy/main_en.html`(目前最完整版本,英文 UI,涵蓋全部模組)——**以此為唯一開發基礎**

`main_en.html` 已經涵蓋 `fragment_zh.html` 的所有功能並重新實作,兩者不需要合併。你的任務是修正 `main_en.html` 裡的問題並補完缺失邏輯,而不是從零重寫。

## 1. 三條不可逆的架構決策(不要重新討論,直接照做)

1. **離線模式 = 預先同步包,不是真零網路。** 任何需要外部資料(道路圖、建築高度、TLE 軌道資料)的模組,都假設「出隊前已下載好快取包」,程式碼裡要明確標出「此處讀取的是預載快取,不是即時網路請求」的註解或 TODO,不要偷偷加 fetch() 去打外部 API。
2. **不做前端直接爬 Telegram / X(Twitter)。** Module D 的「群眾回報衝突偵測」邏輯,資料來源永遠是「人工輸入的回報旗標」(目前程式碼裡 `crowdsourcedAnomaly: true/false` 就是這個概念的資料層),你要做的是把它從「寫死在測試資料裡」變成「有一個輸入表單讓使用者手動標記」,不要實作任何對外爬蟲或社群媒體 API 呼叫。
3. **資料缺值要有退場機制,不能讓模組整個壞掉。** 任何依賴外部資料但目前沒有真實資料的功能(尤其是 SGP4/TLE),寧可明確顯示「資料未載入,顯示為佔位內容」,也不要用寫死的假數字偽裝成已經算出來的結果。

## 2. 本次整合評估抓出的問題清單(逐項驗收標準)

### 🔴 P0 — 必須在合併到 main 之前修正(目前是「假裝有功能」)

**P0-1:Module C 路徑規劃完全是寫死的結果,不是 Dijkstra/A***
- 現況:`workerScript` 裡 `pathTrace = [nodes[3], nodes[1], nodes[0], nodes[2]]` 是常數陣列,不讀取座標或權重。`gridGraph`(含 `weightModifier`)算出來後完全沒被傳進 worker。
- 驗收標準:Web Worker 內必須是真實的 Dijkstra 或 A* 實作,輸入為節點圖(含 `gridX`/`gridY` 座標與邊權重),邊成本 = 基礎距離 × 地形權重係數(步兵:封鎖區×100、危險區×50;裝甲車:封鎖區×10、受損區×1,沿用原規格的權重矩陣)。同一張測試圖,切換步兵/裝甲車兩種模式必須得到不同且邏輯合理的路徑(裝甲車應該抄近路穿過受損區,步兵應該繞開)。
- 測試要求:至少寫一個單元測試,構造一個已知最短路徑答案的小型圖,驗證演算法輸出與手算答案一致。

**P0-2:Module D 軌道倒數計時器是假的(`remainingTicks = 14420` 純計時器)**
- 驗收標準:兩個選項任選一個,但必須誠實標示:
  (a) 實作真實 SGP4 軌道外推(輸入:預載的 TLE 快照,bundled 在 repo 裡,標明取得日期),算出下一次衛星過境時間;或
  (b) 如果暫時沒有 TLE 資料源可整合,UI 必須明確顯示「⚠️ 軌道資料未載入,顯示為估計值」,不能讓使用者以為這是即時計算結果。
- 不接受維持現狀(寫死數字但標籤宣稱是 SGP4)。

**P0-3:AFSK 接收端(RX)沒有位元同步與訊息還原**
- 現況:每次 audio callback 獨立判斷一個位元丟進 log,沒有 bit clock 對齊、沒有 preamble 偵測幀頭、沒有把位元組裝回文字。
- 驗收標準:RX 必須能夠完整還原一段由 TX 端送出的測試字串(例如送出 `"VZ2026:TEST"`,RX 端最終要能顯示出一致的文字,不只是原始位元流)。需實作:preamble 偵測抓到幀頭起點、依 BAUD_RATE 做取樣時鐘對齊(可用過取樣 + 多數投票,如原規格建議)、8 bit 組裝成 byte、`TextDecoder` 還原成字串。
- 測試要求:用 TX 產生的音訊 buffer(可以離線生成,不需要真的對著麥克風測試)餵給 RX 邏輯,驗證能正確還原文字。實體電台的抗噪測試列為後續人工測試項目,不在這次自動化驗收範圍內。

### 🟡 P1 — 應在同一輪修正,風險較低但是真 bug

**P1-1:座標編碼/解碼 roundtrip 是斷的**
- 現況:`Math.abs()` 丟棄經度正負號;`latPacked`/`lngPacked` 沒有固定寬度或分隔符,理論上無法正確切分;`decompressBtn` 根本沒有解析座標,只讀了狀態碼。
- 驗收標準:重新設計編碼格式,確保:(1) 保留經緯度正負號(可用一個額外的符號位,或乾脆固定用 signed Int32 的十六進位表示,不要用 `Math.abs`);(2) 每個欄位固定寬度或有明確分隔符,decode 時能無歧義切回原始數字;(3) `decompressBtn` 要把座標解析出來並顯示,不能只解析狀態碼。寫一個 round-trip 單元測試:任取一組建築物資料 encode 再 decode,結果必須跟原始資料完全一致(座標誤差容許在 ±0.00001 度內,因為是 ×100000 量化)。

**P1-2:i18n 字典資料錯誤 + UI 綁定不完整**
- `i18n.en.header` 目前內容是西班牙文,要改成正確的英文(對照原規格表格的 EN 欄位)。
- `setLanguage()` 目前只更新 2 個 DOM 元素。要改成:所有帶有語言敏感文字的元素(按鈕文字、面板標題、placeholder、靜態說明文字)都要綁定到 `i18n[lang]` 字典並在切換語言時全部更新,不能只更新 header 跟一個按鈕。建議用 `data-i18n-key` attribute 標記所有需要翻譯的元素,`setLanguage()` 統一掃描更新,避免之後又漏掉。

**P1-3:CSS 變數命名沒對齊**
- `.panel-title` 用了未定義的 `--neon-cyan`,但 `:root` 定義的是 `--neon-blue`。全域搜尋取代,確保所有 CSS 變數引用都對應到 `:root` 實際宣告的名稱。

### 🟢 P2 — 數值精度問題,可以晚一點處理

**P2-1:Module B 陰影角度/長度是湊出來的公式,不是真實太陽方位角**
- 現況:`shadowAngle = Math.PI * (0.15 + (hr - 8) * 0.07)` 是滑桿小時數線性湊出來的視覺效果,不是真實天文計算。
- 驗收標準:改用標準太陽位置演算法(輸入:緯度、經度、日期、時刻,輸出:太陽方位角與高度角),再用高度角換算陰影長度(`陰影長度 = 建物高度 / tan(太陽高度角)`)。可以用任何輕量、無外部相依、純 JS 可實作的演算法(如 NOAA 簡化公式),不需要到天文台等級精度,但邏輯必須是真的算出來的,不是滑桿位置硬湊的。

## 3. 模組任務拆分(可指派給平行 subagents)

如果你要用多個 subagent 平行處理,建議按以下邊界拆,**互相之間只透過 `data-contracts/*.schema.json` 定義的資料格式溝通,不要互相改對方檔案內的變數結構**:

| Subagent | 負責範圍 | 對應問題 | 風險等級 |
|---|---|---|---|
| `agent-routing` | Module C:Web Worker 內的 Dijkstra/A* 實作 + 單元測試 | P0-1 | field-critical,需人工 review |
| `agent-codec` | Module A 前半:編碼/解碼格式重新設計 + round-trip 測試 | P1-1 | field-critical,需人工 review |
| `agent-modem` | Module A 後半:AFSK RX 位元同步與訊息還原 | P0-3 | 需離線測試驗證,實體電台測試另外排程 |
| `agent-orbit` | Module D:SGP4 或誠實的佔位機制 | P0-2 | 中 |
| `agent-i18n` | i18n 字典修正 + `data-i18n-key` 綁定機制 | P1-2 | 低,適合作為 good first task |
| `agent-visual` | Module B 太陽位置計算 + CSS 變數修正 | P2-1, P1-3 | 低,適合作為 good first task |

`agent-routing` 跟 `agent-codec` 標記為 field-critical 是因為它們的輸出直接影響「人在哪裡」「該往哪走」這兩個救援決策,合併前需要人工複核測試結果,不要自動合併。

## 4. 明確禁止事項

- 不要實作任何對 Telegram Bot API、X(Twitter)API 的直接呼叫,即使技術上可行(這違反零伺服器的架構決策)。
- 不要在沒有真實 TLE 資料的情況下,用任何形式的隨機數或公式偽裝出一個「看起來合理」的衛星過境時間——沒有資料就要顯示沒有資料。
- 不要為了通過測試而簡化 Dijkstra/A* 成「永遠選某一條固定路徑」——驗收標準是「不同輸入圖會得到不同且邏輯正確的路徑」,不是「程式碼能跑」。
- 完成 P0 項目後,在 PR 描述裡明確列出你修正了哪些測試案例、測試結果是什麼,不要只寫「已修正」。

## 5. 完成定義(Definition of Done)

這次工作完成的標準是:P0 三項各有至少一個能重現「修正前會失敗、修正後會通過」的自動化測試;P1 三項修正完成且 round-trip 測試通過;P2 兩項至少完成其中之一。所有修改都要在對應的 `modules/*` 子資料夾下,不要把不同模組的邏輯寫進同一個檔案——這是為了讓之後開源協作時,貢獻者可以只看懂自己負責的那一塊。
